# LinkedIn Post — ERP → AX Transformation

> 작성일: 2026.04.15
> 대상: 개발자/CTO/테크리드 + 디지털 전환 담당자
> 분량: ~2,800자
> 시리즈: 첫 게시물 (이후 후속 게시물 예정)
> 톤: 절제된 기술 노트 (이모지·임의 구분선 미사용)

---

건설사 ERP를 개발하는 과정에서 AI 에이전트를 함께 도입하면서 AX(AI Transformation)의 성격에 대해 한 가지 정리에 도달했습니다. 거대한 모델을 새로 도입하는 일이라기보다, 사람이 매번 반복하는 의사결정을 단위별로 에이전트에 위임하는 작업에 가깝다는 점입니다.

완성된 시스템에 사후적으로 적용한 것이 아니라 개발과 동시에 두 종류의 Claude 기반 에이전트를 구축해 왔습니다. 하나는 운영 안정성을 다루고, 다른 하나는 개발 속도를 다룹니다.

[1. 핫픽스 자동화 파이프라인 (운영 중)]

운영 환경에서 발생한 에러를 사람이 새벽에 직접 처리하지 않도록 구성한 반응형 에이전트입니다.

가동 정책. Claude Scheduler가 평일 야간과 주말에 한정해 1시간 단위로 Sentry를 폴링합니다. 업무 시간에는 사람이 우선이므로 자동 실행이 멈춥니다.

에러 분류와 행동 지침. 스택트레이스를 분석해 DB 스키마 이슈, 코드 결함, 혼합 케이스 세 가지로 자동 분류합니다. DB 이슈는 ALTER SQL 형태로 Slack에 직접 전달하고, 코드 결함은 hotfix 브랜치에서 코드 수정과 함께 PR을 자동 생성합니다. 즉, 시스템 오류는 해결책을 제시하고 코드 오류는 직접 수정하는 방향으로 행동을 분리했습니다.

도메인 분리. 단일 에이전트가 ERP 전체를 분석하면 컨텍스트 비용과 오진단이 늘어납니다. 도메인별 서브에이전트(Entity 관계, 비즈니스 규칙, 자주 발생하는 에러 패턴 정의)를 별도 sub_agent 문서로 두어 각자 담당 영역만 정밀 분석하도록 구성했습니다.

PR 워크플로. Claude가 PR을 생성하면 Codex가 1차 자동 리뷰를 수행하고, 개발자가 최종 검증한 뒤 dev로 머지됩니다. main 머지 시점에는 Claude Security 리뷰어가 한 번 더 점검합니다.

GitHub 권한 경계. 자동화의 안전 경계는 git 정책으로 강제되어 있습니다. Claude는 dev에서 분기한 hotfix 브랜치 생성과 PR 등록까지만 가능하며, dev 직접 푸시와 main 직접 머지는 차단되어 있습니다.

Sentry 중복 처리. 이미 처리한 이슈는 assignedTo 메타데이터로 마킹하여 중복 보고를 방지합니다. 같은 에러는 재발 시점에만 다시 감지되도록 설계했습니다.

Slack 메시지 표준. 사용자 단위로 양식이 고정되어 있습니다. 임의 제목·임의 접두사를 만들지 않고 DB·코드·혼합 세 가지 양식을 그대로 따르도록 정책에 명시했습니다.

환경 변수는 로컬 설정 파일에 격리해 git에 노출되지 않도록 분리했습니다.

전체 흐름은 단순합니다. API 서버에서 에러가 발생하면 Sentry로 전송되고, Claude Code 스케줄러가 이를 처리한 뒤 결과를 Slack으로 전달합니다. 이 파이프라인이 자리 잡은 덕분에 개발 중인 시스템임에도 야간·주말에 개발자가 호출되는 빈도가 거의 사라졌습니다.

[2. 실시간 코드 리뷰어 (구축 중)]

개발 중인 ERP는 도메인 폭이 넓어 신규 합류자가 코드 구조를 파악하는 데 한 달 가까이 소요되곤 합니다. 또한 같은 코드를 바라보는 시선이 직무별로 갈라집니다. 기획자는 비즈니스 가치를, 개발자는 아키텍처 결정을, 클라이언트는 사용자 경험을 우선합니다.

서브에이전트로 분석, 단일 에이전트로 응답. 코드와 문서 파악 단계에서 개발자·기획자·클라이언트 세 관점을 가진 서브에이전트를 동시에 활용해 도메인 컨텍스트를 정리했습니다. 사용자에게 노출되는 응답은 세 관점을 한 화면에 통합한 형태로 제공됩니다. 메뉴명 한 번을 입력하면 됩니다.

도메인 분류. 분류 기준은 Swagger 태그(10개)이며, 각 도메인의 비즈니스 가치·아키텍처 결정·사용자 시나리오를 별도 sub_agent 문서로 관리합니다. 메뉴와 도메인 매핑은 에이전트 정책 파일에 명시되어 있습니다.

자동 갱신 트리거. 도메인 컨텍스트는 main 브랜치를 기준으로 합니다. main 변경 시 GitHub Actions가 변경된 도메인을 식별하고, 해당 sub_agent 문서를 자동으로 재생성하도록 설계 중입니다. 사람이 손으로 동기화할 일을 없애는 것이 목표입니다.

행동 정책. 리뷰어는 코드 수정을 수행하지 않습니다. Read 계열 도구만 허용되며, 추측성 답변을 막기 위해 인용하는 파일과 라인은 실제 확인을 거친 결과만 노출하도록 정책에 명시되어 있습니다. 응답 형식과 검증 체크리스트도 동일한 정책 파일에 함께 정의됩니다.

범위 분리. 에이전트는 두 범위로 분리해 관리합니다. Global 범위(거버넌스 레포)에는 도메인 내용을 제거한 인터페이스만 둡니다. Project 범위(프로젝트 레포)에는 도메인 내용을 포함한 데이터를 두고 버전을 관리합니다. 이 분리 덕분에 다른 프로젝트로 동일한 구조를 이식할 때 도메인 데이터만 새로 채우면 됩니다.

[설계 원칙 정리]

첫째, 안전 경계는 코드 레벨에서 강제합니다. PR 생성은 자동, 머지는 사람. 이 정책은 git hook과 브랜치 보호 규칙으로 명문화되어 있어 자동화의 임의 권한 확장이 차단됩니다.

둘째, 응답 정확도는 모델 크기보다 컨텍스트 품질에 의해 결정됩니다. 도메인 sub_agent 문서가 사실상 에이전트의 두뇌이며, 문서의 깊이가 응답 품질의 상한을 정합니다.

셋째, 정책과 데이터를 분리합니다. 정책(에이전트 행동 규칙)과 도메인 데이터(프로젝트별 컨텍스트)를 같은 파일에 두면 재사용이 어려워집니다. 두 자산을 서로 다른 SSOT로 분리하는 구조는 멀티 프로젝트 확장을 전제로 한 선택입니다.

[열린 논의: 컨텍스트 저장소를 어디에 둘 것인가]

두 에이전트를 함께 운영하면서 가장 먼저 부딪힌 공통 문제는 한 가지였습니다. 에이전트가 참조하는 컨텍스트 문서를 어디에서 관리할 것인가입니다.

현재 구조는 도메인 sub_agent 문서를 프로젝트 레포에 두고 git으로 버전을 관리합니다. 이 방식의 장점은 변경 이력이 PR 단위로 추적되고 코드 리뷰 흐름과 자연스럽게 결합된다는 점입니다. 그러나 한계가 분명합니다. Claude Code와 Codex는 서로 다른 세션에서 동작하므로 컨텍스트가 자동으로 공유되지 않고, 한쪽이 갱신한 내용을 다른 쪽이 인지하는 시점이 어긋나면 동일 코드에 대해 일관성 없는 응답이 나옵니다.

선택지는 세 가지로 정리됩니다.

레포 기반(현재). git 이력과 PR 리뷰 흐름과 호환되며 진입 비용이 낮습니다. 다만 세션 간 동기화는 pull 시점에 의존하므로 실시간성이 약합니다.

외부 스토리지 기반(S3·GCS 등). 모든 에이전트가 단일 출처를 바라보게 만들 수 있어 일관성 측면에서 유리합니다. 대신 변경 이력 관리, 권한 모델, 캐시 무효화 정책을 별도로 설계해야 합니다.

DB 기반. 빠른 조회와 트랜잭션 일관성을 얻을 수 있고 메타데이터 인덱싱이 쉬워집니다. 단, markdown 본연의 가독성과 PR 단위 리뷰 흐름이 약해지는 트레이드오프가 있습니다.

현실적으로는 정책 인터페이스는 레포에 두고 도메인 컨텍스트 데이터는 외부 스토리지로 분리하는 하이브리드 구조를 검토하고 있습니다. 다만 어느 쪽이 장기적으로 더 합리적인지에 대해서는 아직 결론을 내리지 못했습니다. 비슷한 구조를 운영해 보신 분들의 경험을 듣고 싶습니다.

[다음 단계]

세션 한정으로 운영 중인 리뷰어 에이전트를 Anthropic SDK 기반 Slack 봇으로 이관하는 작업을 준비 중입니다. 슬래시 커맨드 한 줄로 도메인 컨텍스트에 접근 가능한 형태가 1차 목표이며, 장기적으로는 핫픽스 에이전트와 통합한 단일 ERP 봇을 지향합니다. 운영 시 PII 노출 방지(주민번호·실명·전화번호 정규식 차단)와 채널·사용자 단위 권한 화이트리스트가 함께 적용될 예정입니다.

[현재까지 측정한 결과]

운영 에러 처리 시간은 30분에서 수 시간 단위였던 작업이 2분에서 10분 사이로 단축되었습니다. 신규 합류자의 도메인 이해 기간은 한 달 단위에서 며칠 단위로 줄었고, 야간 비상 호출은 사실상 사라졌습니다. 코드 리뷰 회의에서 도메인 컨텍스트를 설명하는 데 쓰던 시간 또한 눈에 띄게 감소했습니다.
