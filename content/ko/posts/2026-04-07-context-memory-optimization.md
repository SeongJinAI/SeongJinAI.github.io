## 문제 정의

AI 코딩 에이전트(Claude Code, Cursor 등)를 사용해 장기 프로젝트를 진행하면, 세션이 바뀔 때마다 같은 문제에 부딪힙니다.

```
세션 1: "급여 계산 로직 수정 중... 컨텍스트 가득 참"
세션 2: "이전 세션에서 뭘 했더라? 어디까지 했지?"
세션 3: "이 파일 왜 이렇게 수정했었지? 다시 분석해야 하나?"
```

**AI 에이전트의 컨텍스트 윈도우는 유한합니다.** 아무리 큰 컨텍스트를 가진 모델이라도, 프로젝트가 커지면 모든 정보를 동시에 담을 수 없습니다. 세션이 끝나면 대화 맥락은 사라지고, 다음 세션은 백지에서 시작합니다.

사람이 매번 "지난번에 이것 하다가 저기서 막혔고, 이 파일을 이런 이유로 수정했고..."라고 설명해야 한다면, AI 에이전트를 쓰는 의미가 반감됩니다.

**이 시스템은 AI 에이전트가 세션 간 맥락을 스스로 복구하도록 설계합니다.**

---

## 핵심 설계 — 관심사별 외부 메모리 분리

인간의 기억도 단일 저장소가 아닙니다. 할 일 목록, 업무 노트, 연락처는 각각 다른 곳에 저장합니다. AI 에이전트의 외부 메모리도 같은 원칙을 따릅니다.

**하나의 파일에 모든 것을 담지 않습니다.** 정보의 성격에 따라 최적의 저장소를 선택합니다.

| 정보 유형 | 저장소 | 수명 | 갱신 주체 |
|----------|--------|------|----------|
| **할 일 목록** (뭘 해야 하나?) | Notion | 작업 완료까지 | 사람 + AI |
| **기술 맥락** (어디까지 했나?) | Handoff 파일 | 다음 세션까지 | AI |
| **영구 참조** (Block ID, API 패턴) | Memory 파일 | 반영구 | AI |
| **인증 정보** (토큰, 시크릿) | .mcp.local.json | 반영구 | 사람 |

이 분리가 중요한 이유: **각 저장소의 갱신 주기와 접근 패턴이 다르기 때문입니다.**

- Notion의 할 일은 **매일** 바뀝니다. 체크하고, 추가하고, 수정합니다.
- Handoff의 기술 맥락은 **세션마다** 덮어씁니다. 이전 세션 끝 → 다음 세션 시작.
- Memory의 Block ID는 **거의 바뀌지 않습니다.** Notion 페이지 구조가 변하지 않는 한 영구적입니다.

이들을 하나의 파일에 합치면, 자주 바뀌는 정보와 안정적인 정보가 섞여 관리가 어렵고, 매 세션마다 불필요한 정보까지 컨텍스트에 로드해야 합니다.

### 구성 요소별 역할

| 구성 요소 | 역할 | 위치 |
|----------|------|------|
| **Skill System** | 명령어 정의 (add/done/list/pending) | `~/.claude/skills/` |
| **Handoff Files** | 세션 간 기술 맥락 전달 | `handoff/{session}.md` |
| **Memory Files** | Block ID, API 패턴 등 영구 참조 | `~/.claude/projects/{project}/memory/` |
| **Secret Store** | Notion 토큰 (git-ignored) | `.claude/.mcp.local.json` |
| **Notion** | 작업 상태 SSOT (Single Source of Truth) | 외부 SaaS |

## 왜 Notion인가 — 외부 SSOT의 조건

AI 에이전트의 외부 메모리로 Notion을 선택한 이유는 **"사람과 AI가 동시에 사용할 수 있는 SSOT"**이기 때문입니다.

마크다운 파일에 체크리스트를 관리할 수도 있습니다. 하지만:

| 방식 | 사람 접근 | AI 접근 | SSOT |
|------|----------|---------|------|
| 마크다운 파일 | 에디터 필요 | ✅ 직접 읽기/쓰기 | ❌ git 충돌 가능 |
| Notion | ✅ 브라우저/앱 | API 호출 필요 | ✅ 항상 최신 |
| Jira/Linear | ✅ 브라우저 | API 호출 필요 | ✅ 항상 최신 |

Notion을 선택한 것은 이미 기획자와 소통하는 채널이었기 때문입니다. 같은 원칙은 Jira, Linear, GitHub Issues 등 어떤 외부 도구에도 적용됩니다.

핵심은 도구가 아니라 **"작업 상태는 외부 SSOT에, 기술 맥락은 로컬 파일에"**라는 분리 원칙입니다.

## 상세 처리 흐름

### 흐름 1: 세션 시작 — 맥락 자동 복구

새 세션이 시작되면 AI 에이전트는 두 소스를 결합하여 현재 상태를 파악합니다.

```
사용자: "이어서 진행합니다"
  │
  ├─ 1. handoff/main.md 읽기
  │     └─ "dev 브랜치, 급여 계산 로직 수정 완료, Lambda 반영 필요"
  │
  ├─ 2. /notion pending 실행
  │     └─ "급여관리 7건, 인사관리 5건, 설계/인프라 6건 미완료"
  │
  └─ 3. 종합
        "급여 계산은 완료됐고, 다음은 급여 정산 salaryType 검증 추가.
         Lambda 4건은 AWS 콘솔 반영 필요 (코드 작업 아님)."
```

**사람이 설명할 필요가 없습니다.** AI가 스스로 맥락을 복구합니다.

### 흐름 2: `/notion pending` — 미완료 작업 조회

이 명령어가 실행될 때 내부적으로 일어나는 일:

```
"/notion pending"
  │
  ├─ 1. Skill 로딩
  │     └─ notion.md가 프롬프트로 주입 (명령어 해석 규칙)
  │
  ├─ 2. Memory 참조
  │     └─ reference_notion_workspace.md에서 Block ID 매핑 획득
  │        ├─ 공사관리: 329c97b4...16e3
  │        ├─ 경비관리: 329c97b4...9cb1
  │        ├─ 급여관리: 329c97b4...2631
  │        └─ ... (8개 메뉴)
  │
  ├─ 3. Token 획득
  │     └─ .mcp.local.json에서 NOTION_TOKEN 읽기
  │
  ├─ 4. 1차 API 호출 (병렬, 8개 메뉴)
  │     └─ GET /blocks/{toggle_id}/children
  │        → to_do 블록 중 checked == false 필터링
  │
  ├─ 5. 2차 API 호출 (하위 toggle)
  │     └─ 경비관리(3개), 급여관리(3개) 하위 메뉴 재귀 조회
  │
  └─ 6. 출력
        [pending] 전체 미완료 항목 (31건)
          📌 공사관리 (4건)
            [ ] 공정관리
            [ ] DB > [레미콘타설현황] > 수정 > ...
          📌 급여관리 (8건)
            [ ] API > [급여계산] > 개선 > ...
```

**Block ID를 매번 찾지 않습니다.** Memory에 한 번 저장하면 이후 모든 세션에서 재사용합니다.

### 흐름 3: 세션 종료 — 맥락 저장

```
사용자: "/clear"
  │
  ├─ 1. handoff/{session}.md 업데이트
  │     ├─ 완료 작업 기록
  │     ├─ 브랜치/빌드 상태
  │     ├─ 주의사항
  │     └─ 다음 세션 시작 프롬프트
  │
  ├─ 2. handoff/INDEX.md 갱신
  │     └─ 마지막 업데이트 날짜
  │
  └─ 3. 컨텍스트 해제
        "인수인계 완료 (세션: main)"
```

**미완료 작업은 Handoff에 기록하지 않습니다.** 그것은 Notion의 역할입니다. Handoff에는 "기술적으로 알아야 할 것"만 남깁니다.

## 멀티 세션 — 병렬 작업 맥락 분리

프로젝트가 커지면 하나의 세션으로 모든 작업을 처리할 수 없습니다. 기능 개발 중에 긴급 버그가 들어오고, 인프라 설정을 변경해야 할 때도 있습니다.

```
handoff/
├── INDEX.md          ← 세션 라우팅 규칙
├── main.md           ← 기능 개발 (dev 브랜치)
├── hotfix.md         ← 긴급 버그 (hotfix/* 브랜치)
└── infra.md          ← 인프라/자동화
```

각 세션은 **독립된 기술 맥락**을 가집니다.

| 규칙 | 이유 |
|------|------|
| 자기 파일만 쓰기 | 세션 간 맥락 오염 방지 |
| 다른 파일은 읽기만 | 관련 작업 참조는 허용 |
| 브랜치로 자동 라우팅 | `hotfix/*` → hotfix.md |

이 구조가 없으면 AI 에이전트가 "급여 계산 수정 중인데 Sentry 에러 처리해야 해서 브랜치를 바꿨고, 다시 돌아왔는데 어디까지 했더라..."라는 상황에 빠집니다.

## MCP가 안 될 때 — 우회 설계

Claude Code는 MCP(Model Context Protocol)로 외부 도구와 연동합니다. 하지만 현실에서는 MCP가 항상 작동하지 않습니다.

**실제 문제:** 내장 Notion MCP가 다른 워크스페이스의 OAuth에 연결되어, 업무용 Notion에 접근 불가.

```
이상적 흐름:
  Claude Code → MCP → Notion (OAuth) → ✅

실제 상황:
  Claude Code → MCP → 다른 워크스페이스 OAuth → ❌ 접근 불가

우회 설계:
  Claude Code → Skill (curl 직접 호출) → Notion API → ✅
```

MCP에 의존하지 않고 **curl + REST API + Integration Token**으로 직접 호출합니다. MCP가 정상화되면 전환할 수 있지만, 현재는 이 방식이 더 안정적입니다.

## 인터페이스-구현 분리 — 프로젝트 간 재사용

이 시스템은 특정 프로젝트에 종속되지 않습니다. **거버넌스 레포에서 인터페이스를 정의**하고, 각 프로젝트에서 구현을 오버라이딩합니다.

```
거버넌스 레포 (인터페이스)              프로젝트 (구현)
┌─────────────────────┐          ┌──────────────────────┐
│ skills/              │          │ .claude/skills/       │
│   intg-notion.md     │ ──────→ │   notion.md           │
│   "add/done/list/    │ install │   "메뉴→Block ID 매핑 │
│    pending 명령 정의" │          │    토큰 위치 정의"     │
└─────────────────────┘          └──────────────────────┘
```

새 프로젝트에서 이 시스템을 쓰려면:

1. `install.sh` 실행 → 스킬 인터페이스 복사
2. 프로젝트 `.claude/skills/notion.md`에 Block ID 매핑 작성
3. `.mcp.local.json`에 Notion 토큰 설정

**명령어 양식, 출력 형식, 에러 처리는 거버넌스에서 관리**하므로 프로젝트마다 다시 정의할 필요가 없습니다.

## 개발자 업무 개선 효과

### 정량적 개선

| 지표 | 기존 | 자동화 후 | 개선율 |
|------|------|----------|-------|
| 세션 시작 맥락 복구 | 5~10분 (대화로 설명) | **30초** (`/notion pending`) | **90%+** |
| 작업 목록 확인 | Notion 브라우저 열기 + 스크롤 | **명령어 한 줄** | 즉시 |
| 세션 종료 인수인계 | 수동 정리 또는 생략 | **자동 생성** (`/clear`) | 누락 방지 |
| 다음 세션 첫 질문 | "지난번에 뭘 했었지?" | **불필요** | 제거 |

### 정성적 개선

- **컨텍스트 낭비 제거** — 이전 대화 내용을 다시 설명하는 데 컨텍스트를 쓰지 않음
- **멀티 세션 안전** — 기능 개발 ↔ 핫픽스 전환 시 맥락이 섞이지 않음
- **기획자 소통 통합** — 같은 Notion 페이지에서 기획자는 요구사항을 추가하고, AI는 완료 체크

## 기술 스택

| 계층 | 도구 | 선택 이유 |
|------|------|----------|
| 작업 관리 | **Notion** | 기획자-개발자 공유 SSOT, REST API 제공 |
| AI 에이전트 | **Claude Code** | Skill/Memory/Hook 확장 시스템 |
| 명령 인터페이스 | **Skill (.md)** | 마크다운으로 명령어 정의, 프롬프트 주입 |
| 영구 기억 | **Memory (.md)** | 대화 간 지속, 자동 로드 |
| 세션 맥락 | **Handoff (.md)** | 세션별 기술 컨텍스트 파일 분리 |
| 토큰 관리 | **.mcp.local.json** | git-ignored, 프로젝트 로컬 |
| 거버넌스 | **claude-dotfiles** | 멀티 프로젝트 인터페이스 관리 |

## 현재 한계 및 로드맵

### 현재 한계

| 한계 | 원인 | 영향 |
|------|------|------|
| Notion API 순차 호출 | 하위 toggle 재귀 조회 필요 | 8개 메뉴 + 하위 6개 = 14회 API 호출 |
| Block ID 수동 매핑 | Notion 페이지 구조 변경 시 메모리 업데이트 필요 | 메뉴 추가/삭제 시 수작업 |
| MCP 미사용 | OAuth 워크스페이스 불일치 | curl 직접 호출로 우회 중 |

### 로드맵

| 단계 | 개선 | 효과 |
|------|------|------|
| Phase 2 | Block ID 자동 탐색 (페이지 구조 파싱) | 메뉴 변경 시 자동 대응 |
| Phase 3 | MCP 정상화 시 네이티브 전환 | curl 우회 제거 |
| Phase 4 | 양방향 동기화 (코드 변경 → Notion 자동 체크) | 수동 `/notion done` 제거 |
