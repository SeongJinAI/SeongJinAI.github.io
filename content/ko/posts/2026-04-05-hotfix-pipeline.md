## 문제 정의

서버 운영 중 에러가 발생하면 개발자는 매번 동일한 사이클을 반복합니다.

```
알림 확인 → 서버 접속 → 로그 분석 → 원인 파악 → 수정 → 테스트 → PR → 리뷰 → 배포
```

에러 하나를 처리하는 데 **30분에서 수 시간**이 소요됩니다. 더 큰 문제는 에러의 절반 이상이 DB 스키마 불일치처럼 **코드 수정이 필요 없는 단순한 문제**임에도 동일한 프로세스를 거쳐야 한다는 점입니다.

이 시스템은 위 전체 프로세스를 AI가 자동으로 수행합니다.

---

## 핵심 설계 — 에러 유형별 자동 분기

모든 에러를 같은 방식으로 처리하지 않습니다. 에러 메시지를 분석하여 **유형을 자동으로 판별**하고, 유형에 따라 완전히 다른 대응 경로를 타게 합니다.

| 유형 | 판별 기준 | 대응 | 개발자 행동 |
|------|----------|------|-----------|
| **DB 에러** | 컬럼 누락, 타입 불일치, 스키마 오류 | Slack으로 수정 SQL 직접 전달 | SQL 실행만 |
| **코드 에러** | NPE, 로직 버그, 검증 누락 | AI가 코드 수정 후 PR 생성 | 계획 승인 + PR 머지 |
| **혼합** | 코드와 DB 양쪽 불일치 | SQL(Slack) + PR(코드) 동시 | SQL 실행 + PR 머지 |

이 분기가 중요한 이유: **DB 에러는 코드를 아무리 수정해도 해결되지 않습니다.** 기존에는 개발자가 코드부터 확인하고, DB 문제임을 뒤늦게 깨닫는 시간 낭비가 있었습니다. 이 시스템은 에러 메시지 패턴만으로 즉시 판별합니다.

         │  (주기적 감시)         │
         │                       │
         │  ① 새 에러 감지        │
         │  ② 스택트레이스 분석   │
         │  ③ 코드베이스 탐색     │

         └───────┬───────────────┘
                 │
        ┌────────┼────────┐
        ▼        ▼        ▼
   ┌────────┐ ┌──────┐ ┌──────┐

   └───┬────┘ └──┬───┘ └──┬───┘

   ┌────────┐ ┌──────┐ ┌──────────┐

## 도메인 전문 서브에이전트 — 에러 분석의 정확도를 높이는 구조

하나의 AI 에이전트가 시스템 전체를 분석하면 컨텍스트 낭비와 오진이 발생합니다. 이 시스템은 **도메인별 전문 서브에이전트**를 두어 에러가 발생한 영역의 전문가에게 분석을 위임합니다.

### 서브에이전트 라우팅

에러의 스택트레이스에서 패키지 경로를 추출하고, 해당 도메인의 서브에이전트를 자동으로 호출합니다.

```
메인 에이전트 (Hotfix Pipeline)
    │
    ├─ 스택트레이스에서 도메인 판별
    │
    ├─ 서브에이전트 호출 ──→ Domain A 에이전트
    │                      "이 에러를 네 도메인 지식으로 분석해"
    │                         ├─ Entity 관계 파악
    │                         ├─ 비즈니스 규칙 검증
    │                         ├─ 에러 패턴 매칭
    │                         └─ 상세 문서 참조 (명세서/아키텍처)
    │
    ├─ 분석 결과 수신
    │
    └─ 유형 판별 → DB/코드/혼합 → 대응
```

### 서브에이전트 패키지 구조

메인 에이전트 내부에 도메인별 컨텍스트 파일을 패키지로 관리합니다. 각 파일에는 해당 도메인의 **데이터 모델, 비즈니스 규칙, API 목록, 에러 패턴, 기술 문서 참조 경로**가 정의되어 있습니다.

```
hotfix-pipeline/
├── AGENT.md              ← 메인 (도메인 라우팅 + 에러 유형 분류)
└── domains/
    ├── domain-a.md       ← 도메인 A 컨텍스트
    ├── domain-b.md       ← 도메인 B 컨텍스트
    ├── domain-c.md       ← 도메인 C 컨텍스트
    └── ...               ← 시스템 규모에 따라 확장
```

이 구조는 **거버넌스 레포에서 양식(인터페이스)만 제공**하고, 각 프로젝트에서 실제 도메인 지식을 채웁니다. 새 프로젝트에 적용할 때 패키지 구조를 복사하고 도메인 파일만 작성하면 됩니다.

### 왜 서브에이전트인가?

| 단일 에이전트 | 서브에이전트 |
|-------------|------------|
| 시스템 전체를 컨텍스트에 로드 | 해당 도메인 지식만 로드 |
| 컨텍스트 제한으로 핵심 정보 유실 가능 | 도메인에 집중하여 정밀 분석 |
| 도메인 간 규칙 혼동 가능 | 독립 실행으로 혼동 없음 |
| 모든 에러에 동일한 분석 깊이 | **도메인별 맞춤 분석** |

도메인이 다수이고 각각 수십 개의 Entity와 비즈니스 규칙을 가진 엔터프라이즈 환경에서는, 서브에이전트 없이 정확한 분석이 사실상 불가능합니다.

## 에러 유형별 처리 상세

### Type A: DB 에러 — SQL 직접 전달 (코드 수정 없음)

DB 레벨 문제는 코드 변경으로 해결되지 않습니다. AI가 애플리케이션 스키마 정의와 실제 DB 상태를 비교 분석하여 정확한 수정 SQL을 생성합니다.

**자동 판별:**
- "Unknown column" → 컬럼 누락
- "Data too long" → 타입/크기 불일치
- "Table doesn't exist" → 테이블 미생성

**처리 흐름:**
```
에러 감지 → 스키마 정의 vs DB 비교 → ALTER SQL 생성 → Slack 전달 → 개발자가 SQL 실행
```

DB 에러는 PR이 불필요하므로 **Slack 메시지 하나로 해결**됩니다. 기존에 SSH 접속 → 로그 분석 → SQL 작성에 30분 걸리던 것이 **2분 이내**로 단축됩니다.

### Type B: 코드 에러 — 자동 분석 + PR 생성

코드 레벨 버그는 AI가 스택트레이스를 따라 원인을 파악하고, 수정 계획을 세운 후 개발자 승인을 받아 PR을 생성합니다.

**처리 흐름:**
```
에러 감지 → 스택트레이스 추적 → 코드 분석 → 수정 계획 작성 → Slack 공유
→ 개발자 승인 → 브랜치 생성 → 코드 수정 → PR → 코드 리뷰 → 리뷰 반영
→ 개발자 최종 머지
```

### Type C: 혼합 — SQL + PR 동시

애플리케이션 정의와 DB가 동시에 불일치하는 경우, 양쪽 수정을 병렬로 처리합니다.

## 설계 철학 — SOTT (Scoped One-Time Task)

이 시스템의 근본 원칙: **AI는 분석하고 제안하지만, 최종 결정은 항상 개발자가 내린다.**

AI에게 무제한 권한을 부여하지 않습니다. 에러가 감지될 때마다 **"이 에러에 대해서만, 이 범위에서만"** 제한된 권한을 부여합니다.

```
  에러 발생 → AI 자동 분석 (권한 불필요, 읽기만)
                    ↓
         개발자 승인 (SOTT 부여)
         "이 에러를 이 방법으로 수정 허가"
         범위: 특정 에러 1건
         권한: hotfix 브랜치만
         만료: PR 생성 시
                    ↓
         AI 자동 수정 (승인된 범위 내에서만)
                    ↓
         개발자 최종 머지 결정
```

### 왜 완전 자동화가 아닌가?

DB SQL 실행과 코드 머지는 **되돌리기 어려운 작업**입니다. AI가 99% 정확하더라도 1%의 오판이 운영 데이터를 손상시킬 수 있습니다.

- **분석과 계획은 자동** — AI가 가장 잘하는 영역
- **실행 결정은 사람** — 되돌릴 수 없는 작업의 최종 게이트
- **반복 작업은 자동** — 브랜치 생성, 커밋, PR, 리뷰 반영

### 개발자 개입 포인트 (3곳만)

| # | 지점 | SOTT 범위 | 소요 시간 |
|---|------|----------|----------|
| 1 | **수정 계획 승인** | 이 에러를 이 방법으로 수정 | 1분 |
| 2 | **DB SQL 실행** | 이 SQL을 운영 DB에 실행 | 1분 |
| 3 | **PR 최종 머지** | 이 코드를 운영에 반영 | 2분 |

나머지는 **전부 자동**입니다. 개발자는 **판단만** 하고, 실행은 AI가 합니다.

## 개발자 업무 개선 효과

### 정량적 개선

| 지표 | 기존 | 자동화 후 | 개선율 |
|------|------|----------|-------|
| DB 에러 평균 처리 시간 | 30분 | **2분** | **93%** |
| 코드 에러 평균 처리 시간 | 1~2시간 | **10분** | **83~91%** |
| 야간/주말 초기 대응 | 다음 영업일 | **즉시** | 대응 공백 제거 |

### 정성적 개선

- **컨텍스트 스위칭 최소화** — 에러 분석을 AI가 수행하므로 개발자는 기능 개발에 집중
- **DB vs 코드 자동 판별** — 잘못된 방향으로 디버깅하는 시간 제거
- **일관된 수정 품질** — 프로젝트 코드 컨벤션을 준수한 자동 수정
- **이력 자동 관리** — 모든 에러 분석/수정 이력이 자동 기록

## 이중 모니터링 — 원격 감시병 + 로컬 실행자

에러 모니터링을 단일 포인트에 의존하지 않습니다. **원격(24시간)과 로컬(세션 중)**을 이중으로 운영합니다.

### 원격 트리거 (Anthropic 클라우드)

매 1시간마다 새 샌드박스가 생성되고, GitHub에서 레포를 clone한 뒤 프롬프트를 실행합니다. **서버리스 함수(AWS Lambda)와 동일한 개념**입니다. 이전 실행의 상태를 기억하지 않습니다.

```
매 정시:
  새 샌드박스 → GitHub clone → Sentry API 조회
  → 에러 발견 → 도메인 서브에이전트 파일 참조 → 분석
  → DB 에러: Slack SQL 전달
  → 코드 에러: hotfix 브랜치 → PR 생성 → Slack PR 링크
  → 폐기
```

**업무시간 정책**: 평일 09:00~18:00 KST는 개발자가 직접 모니터링하므로 원격이 자동 스킵합니다.

### 로컬 폴백 (개발자 PC)

```bash
/loop 10m .claude/scripts/sentry-check.sh
```

세션이 열려 있는 동안 10분 간격으로 Sentry를 체크합니다. 원격과 달리 **DB 접근, 로컬 파일 접근, 즉시 에이전트 호출**이 가능합니다.

| | 원격 (감시병) | 로컬 (실행자) |
|--|:-----------:|:-----------:|
| 주기 | 1시간 | 10분 |
| 지속성 | 24시간 | 세션 중만 |
| Sentry 분석 | ✅ | ✅ |
| 도메인 지식 | ✅ (GitHub clone) | ✅ (로컬 파일) |
| Slack 전달 | ✅ | ✅ |
| PR 생성 | ✅ (hotfix→dev) | ✅ (hotfix→dev) |
| DB 접근 | ❌ | ✅ |

### 향후 개선: Push 기반 즉시 감지

현재는 Polling(주기적 조회) 방식입니다. 즉시 감지하려면 **Sentry Webhook → AWS Lambda → Claude RemoteTrigger run API** 구조가 필요합니다. Lambda가 Sentry의 HTTP 요청을 받아 Claude 원격 에이전트를 깨우는 중간 다리 역할입니다. 인프라 추가가 필요하므로 다음 단계로 계획 중입니다.

## 운영 정책 — 에이전트가 따르는 규칙

원격 트리거는 매 실행마다 **독립 환경(stateless)**입니다. 이전 실행의 상태를 기억하지 않습니다. 이런 환경에서 안정적으로 동작하려면 명확한 운영 정책이 필요합니다.

### 프롬프트-문서 분리 아키텍처

트리거 프롬프트에 모든 규칙을 넣으면 수정할 때마다 API를 호출해야 합니다. 대신 프롬프트는 **"AGENT.md를 읽고 따르라"**는 지시만 담고, 상세 정책은 레포의 문서 파일에 관리합니다.

```
trigger-prompt.md (프롬프트 원본, git 추적)
    │
    └─ "AGENT.md를 먼저 읽어라"
         │
         ├─ 에이전트 실행 규칙 (MANDATORY)
         ├─ 에러 유형 판별 정책
         ├─ Slack 메시지 템플릿
         ├─ 중복 보고 방지 정책
         └─ domains/*.md (도메인 지식)
```

정책 변경 시 **레포의 md 파일만 수정**하면 다음 실행에서 자동 반영됩니다. 트리거 재배포가 필요 없습니다.

### 9단계 실행 체크리스트

에이전트가 이슈를 처리할 때 반드시 따르는 순서입니다.

```
□ 1. Sentry 조회 — 미해결 + 미할당 + prod
□ 2. 최신 이벤트 조회
□ 3. 스택트레이스에서 패키지 경로 추출
□ 4. 에러 유형 판별 (decision tree)
□ 5. 해당 도메인 md 파일 읽기 ← 필수
□ 6. 도메인 컨텍스트 기반 원인 분석
□ 7. 유형별 대응 (DB→SQL / CODE→PR / MIXED→둘 다)
□ 8. Slack 메시지 전송 (템플릿 적용)
□ 9. Sentry 이슈 assign (중복 방지)
```

5번(도메인 md 읽기)이 핵심입니다. 각 도메인 파일에는 Entity 관계, 비즈니스 규칙, 자주 발생하는 에러 패턴이 기록되어 있어서, 이걸 읽어야 정확한 원인 분석이 가능합니다.

### 에러 유형 판별 Decision Tree

```
에러 메시지 분석
│
├─ SQL 키워드? ("Unknown column", "Data too long", "Table doesn't exist")
│   ├─ Entity에 @Column 존재 + DB 컬럼 미존재 → MIXED
│   └─ 그 외 → DB
│
├─ DataIntegrityViolationException?
│   ├─ NOT NULL / UNIQUE 위반 → CODE (검증 누락)
│   └─ 컬럼 타입 불일치 → DB
│
├─ NullPointerException / RuntimeException → CODE
│
└─ 기타 → CODE (기본값)
```

### Slack 메시지 템플릿

유형별로 고정된 형식을 사용합니다. 필수 항목: **심각도, Sentry ID, 에러 원문, 원인 분석, 해결책**.

| 유형 | 이모지 | 핵심 전달 내용 |
|------|:------:|-------------|
| DB | 🔴 | 해결 SQL + Entity/DDL 참조 |
| CODE | 🟡 | 원인 분석 + PR 링크 + 수정 파일 |
| MIXED | 🟠 | SQL(즉시) + PR(필요 시) |

### 중복 보고 방지 — Sentry Assign 전략

Stateless 환경에서 같은 이슈를 매시간 반복 보고하면 노이즈입니다. **Sentry의 assign 기능**을 외부 상태 저장소로 활용합니다.

```
조회: is:unresolved + !is:assigned + environment:prod
       → 아직 처리되지 않은 이슈만 반환

처리: 분석 → Slack 전송 → PR 생성

완료: PUT /issues/{id}/ {"assignedTo": "..."}
       → 다음 실행에서 자동 제외

재발: 개발자가 resolve → 에러 재발 → Sentry 자동 reopen
       → unresolved + unassigned → 다시 감지
```

상태를 저장하지 않는 서버리스 환경에서도 **자연스러운 순환 구조**가 만들어집니다.

### 브랜치 보호 — 자동 수정의 안전장치

AI가 자동으로 코드를 수정할 때 가장 중요한 것은 **실수의 영향 범위를 제한하는 것**입니다.

```
허용: hotfix/{description} 브랜치 생성 → dev PR
차단: dev/main에 직접 push
차단: dev → main PR 생성
```

이 규칙은 **3단계로 강제**됩니다:
1. `CLAUDE.md` — AI가 규칙을 인지
2. `settings.json` Hook — 도구 레벨에서 실행 자체를 차단
3. Memory — 세션이 바뀌어도 규칙 유지

AI가 아무리 확신을 가지고 수정하더라도, PR을 통해 사람이 최종 확인하지 않으면 dev에 반영되지 않습니다.

### 현재 한계

| 한계 | 원인 | 영향 |
|------|------|------|
| 10분 감지 지연 | Polling 방식 (Scheduled Trigger) | P0 에러에 즉시 대응 불가 |
| 승인 대기 | 개발자가 직접 확인 필요 | 부재 시 자동 수정 지연 |

### 로드맵

| 단계 | 개선 | 효과 |
|------|------|------|
| Phase 2 | Webhook 기반 실시간 감지 | Polling 제거, 즉시 대응 |
| Phase 3 | 메신저 버튼으로 승인/거부 | 세션 없이 승인 가능 |
