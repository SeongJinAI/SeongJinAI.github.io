## 문제 정의

서버 운영 중 에러가 발생하면 개발자는 매번 동일한 사이클을 반복합니다.

```
알림 확인 → 서버 접속 → 로그 분석 → 원인 파악 → 수정 → 테스트 → PR → 리뷰 → 배포
```

에러 하나를 처리하는 데 **30분에서 수 시간**이 소요됩니다. 더 큰 문제는 에러의 절반 이상이 DB 스키마 불일치처럼 **코드 수정이 필요 없는 단순한 문제**임에도 동일한 프로세스를 거쳐야 한다는 점입니다.

이 시스템은 위 전체 프로세스를 AI가 자동으로 수행합니다.

---

## 핵심 설계 — 에러 유형별 자동 분기

모든 에러를 같은 방식으로 처리하지 않습니다. 에러 메시지를 분석하여 **유형을 자동으로 판별**하고, 유형에 따라 완전히 다른 대응 경로를 타게 합니다.

| 유형 | 판별 기준 | 대응 | 개발자 행동 |
|------|----------|------|-----------|
| **DB 에러** | 컬럼 누락, 타입 불일치, 스키마 오류 | Slack으로 수정 SQL 직접 전달 | SQL 실행만 |
| **코드 에러** | NPE, 로직 버그, 검증 누락 | AI가 코드 수정 후 PR 생성 | 계획 승인 + PR 머지 |
| **혼합** | 코드와 DB 양쪽 불일치 | SQL(Slack) + PR(코드) 동시 | SQL 실행 + PR 머지 |

이 분기가 중요한 이유: **DB 에러는 코드를 아무리 수정해도 해결되지 않습니다.** 기존에는 개발자가 코드부터 확인하고, DB 문제임을 뒤늦게 깨닫는 시간 낭비가 있었습니다. 이 시스템은 에러 메시지 패턴만으로 즉시 판별합니다.

## 시스템 아키텍처

> 아키텍처 다이어그램은 draw.io로 제작 예정입니다.

### 전체 흐름

```
┌──────────────┐     ┌──────────┐     ┌──────────┐
│   서버       │────→│  Sentry  │────→│  Slack   │
│              │     │ (에러 수집)│     │ (알림)   │
└──────────────┘     └──────────┘     └────┬─────┘
                                           │
                     ┌─────────────────────┘
                     ▼
         ┌───────────────────────┐
         │   AI Agent            │
         │  (주기적 감시)         │
         │                       │
         │  ① 새 에러 감지        │
         │  ② 스택트레이스 분석   │
         │  ③ 코드베이스 탐색     │
         │  ④ 에러 유형 판별      │
         └───────┬───────────────┘
                 │
        ┌────────┼────────┐
        ▼        ▼        ▼
   ┌────────┐ ┌──────┐ ┌──────┐
   │   DB   │ │ 코드 │ │ 혼합 │
   └───┬────┘ └──┬───┘ └──┬───┘
       ▼         ▼         ▼
   ┌────────┐ ┌──────┐ ┌──────────┐
   │ Slack  │ │  PR  │ │Slack+PR  │
   │SQL 전달│ │ 생성 │ │동시 처리  │
   └────────┘ └──┬───┘ └──────────┘
                 ▼
         ┌───────────────┐
         │  자동 코드 리뷰 │
         └───────┬───────┘
                 ▼
         ┌───────────────┐
         │ 개발자 최종    │
         │ 확인 + 머지    │
         └───────────────┘
```

### 구성 요소

| 구성 요소 | 역할 | 기술 |
|----------|------|------|
| 에러 수집 | 서버 예외 캡처, 스택트레이스 보존 | Sentry |
| 알림 | 에러 발생 즉시 개발자 + AI 동시 통보 | Slack Webhook |
| AI 분석 | 스택트레이스 파싱 → 코드 탐색 → 원인 파악 → 유형 판별 | Claude Code |
| 코드 수정 | 프로젝트 컨벤션을 준수한 자동 수정 | Claude Code |
| PR 관리 | 브랜치 생성 → 커밋 → PR → 리뷰 반영 | GitHub CLI |
| 코드 리뷰 | PR 생성 시 자동 리뷰 | GPT Codex |
| 배포 | dev 머지 → 자동 배포 | Docker + CI/CD |

## 도메인 전문 서브에이전트 — 에러 분석의 정확도를 높이는 구조

하나의 AI 에이전트가 시스템 전체를 분석하면 컨텍스트 낭비와 오진이 발생합니다. 이 시스템은 **도메인별 전문 서브에이전트**를 두어 에러가 발생한 영역의 전문가에게 분석을 위임합니다.

### 서브에이전트 라우팅

에러의 스택트레이스에서 패키지 경로를 추출하고, 해당 도메인의 서브에이전트를 자동으로 호출합니다.

```
메인 에이전트 (Hotfix Pipeline)
    │
    ├─ 스택트레이스에서 도메인 판별
    │
    ├─ 서브에이전트 호출 ──→ Domain A 에이전트
    │                      "이 에러를 네 도메인 지식으로 분석해"
    │                         ├─ Entity 관계 파악
    │                         ├─ 비즈니스 규칙 검증
    │                         ├─ 에러 패턴 매칭
    │                         └─ 상세 문서 참조 (명세서/아키텍처)
    │
    ├─ 분석 결과 수신
    │
    └─ 유형 판별 → DB/코드/혼합 → 대응
```

### 서브에이전트 패키지 구조

메인 에이전트 내부에 도메인별 컨텍스트 파일을 패키지로 관리합니다. 각 파일에는 해당 도메인의 **데이터 모델, 비즈니스 규칙, API 목록, 에러 패턴, 기술 문서 참조 경로**가 정의되어 있습니다.

```
hotfix-pipeline/
├── AGENT.md              ← 메인 (도메인 라우팅 + 에러 유형 분류)
└── domains/
    ├── domain-a.md       ← 도메인 A 컨텍스트
    ├── domain-b.md       ← 도메인 B 컨텍스트
    ├── domain-c.md       ← 도메인 C 컨텍스트
    └── ...               ← 시스템 규모에 따라 확장
```

이 구조는 **거버넌스 레포에서 양식(인터페이스)만 제공**하고, 각 프로젝트에서 실제 도메인 지식을 채웁니다. 새 프로젝트에 적용할 때 패키지 구조를 복사하고 도메인 파일만 작성하면 됩니다.

### 왜 서브에이전트인가?

| 단일 에이전트 | 서브에이전트 |
|-------------|------------|
| 시스템 전체를 컨텍스트에 로드 | 해당 도메인 지식만 로드 |
| 컨텍스트 제한으로 핵심 정보 유실 가능 | 도메인에 집중하여 정밀 분석 |
| 도메인 간 규칙 혼동 가능 | 독립 실행으로 혼동 없음 |
| 모든 에러에 동일한 분석 깊이 | **도메인별 맞춤 분석** |

도메인이 다수이고 각각 수십 개의 Entity와 비즈니스 규칙을 가진 엔터프라이즈 환경에서는, 서브에이전트 없이 정확한 분석이 사실상 불가능합니다.

## 에러 유형별 처리 상세

### Type A: DB 에러 — SQL 직접 전달 (코드 수정 없음)

DB 레벨 문제는 코드 변경으로 해결되지 않습니다. AI가 애플리케이션 스키마 정의와 실제 DB 상태를 비교 분석하여 정확한 수정 SQL을 생성합니다.

**자동 판별:**
- "Unknown column" → 컬럼 누락
- "Data too long" → 타입/크기 불일치
- "Table doesn't exist" → 테이블 미생성

**처리 흐름:**
```
에러 감지 → 스키마 정의 vs DB 비교 → ALTER SQL 생성 → Slack 전달 → 개발자가 SQL 실행
```

DB 에러는 PR이 불필요하므로 **Slack 메시지 하나로 해결**됩니다. 기존에 SSH 접속 → 로그 분석 → SQL 작성에 30분 걸리던 것이 **2분 이내**로 단축됩니다.

### Type B: 코드 에러 — 자동 분석 + PR 생성

코드 레벨 버그는 AI가 스택트레이스를 따라 원인을 파악하고, 수정 계획을 세운 후 개발자 승인을 받아 PR을 생성합니다.

**처리 흐름:**
```
에러 감지 → 스택트레이스 추적 → 코드 분석 → 수정 계획 작성 → Slack 공유
→ 개발자 승인 → 브랜치 생성 → 코드 수정 → PR → 코드 리뷰 → 리뷰 반영
→ 개발자 최종 머지
```

### Type C: 혼합 — SQL + PR 동시

애플리케이션 정의와 DB가 동시에 불일치하는 경우, 양쪽 수정을 병렬로 처리합니다.

## 설계 철학 — SOTT (Scoped One-Time Task)

이 시스템의 근본 원칙: **AI는 분석하고 제안하지만, 최종 결정은 항상 개발자가 내린다.**

AI에게 무제한 권한을 부여하지 않습니다. 에러가 감지될 때마다 **"이 에러에 대해서만, 이 범위에서만"** 제한된 권한을 부여합니다.

```
  에러 발생 → AI 자동 분석 (권한 불필요, 읽기만)
                    ↓
         개발자 승인 (SOTT 부여)
         "이 에러를 이 방법으로 수정 허가"
         범위: 특정 에러 1건
         권한: hotfix 브랜치만
         만료: PR 생성 시
                    ↓
         AI 자동 수정 (승인된 범위 내에서만)
                    ↓
         개발자 최종 머지 결정
```

### 왜 완전 자동화가 아닌가?

DB SQL 실행과 코드 머지는 **되돌리기 어려운 작업**입니다. AI가 99% 정확하더라도 1%의 오판이 운영 데이터를 손상시킬 수 있습니다.

- **분석과 계획은 자동** — AI가 가장 잘하는 영역
- **실행 결정은 사람** — 되돌릴 수 없는 작업의 최종 게이트
- **반복 작업은 자동** — 브랜치 생성, 커밋, PR, 리뷰 반영

### 개발자 개입 포인트 (3곳만)

| # | 지점 | SOTT 범위 | 소요 시간 |
|---|------|----------|----------|
| 1 | **수정 계획 승인** | 이 에러를 이 방법으로 수정 | 1분 |
| 2 | **DB SQL 실행** | 이 SQL을 운영 DB에 실행 | 1분 |
| 3 | **PR 최종 머지** | 이 코드를 운영에 반영 | 2분 |

나머지는 **전부 자동**입니다. 개발자는 **판단만** 하고, 실행은 AI가 합니다.

## 개발자 업무 개선 효과

### 정량적 개선

| 지표 | 기존 | 자동화 후 | 개선율 |
|------|------|----------|-------|
| DB 에러 평균 처리 시간 | 30분 | **2분** | **93%** |
| 코드 에러 평균 처리 시간 | 1~2시간 | **10분** | **83~91%** |
| 야간/주말 초기 대응 | 다음 영업일 | **즉시** | 대응 공백 제거 |

### 정성적 개선

- **컨텍스트 스위칭 최소화** — 에러 분석을 AI가 수행하므로 개발자는 기능 개발에 집중
- **DB vs 코드 자동 판별** — 잘못된 방향으로 디버깅하는 시간 제거
- **일관된 수정 품질** — 프로젝트 코드 컨벤션을 준수한 자동 수정
- **이력 자동 관리** — 모든 에러 분석/수정 이력이 자동 기록

## 기술 스택

| 계층 | 도구 | 선택 이유 |
|------|------|----------|
| 에러 수집 | **Sentry** | 스택트레이스 보존, REST API 제공 |
| 알림 | **Slack** | 개발자가 상주하는 채널, 양방향 소통 |
| AI 엔진 | **Claude Code** | 대규모 컨텍스트, 코드 분석과 수정 동시 가능 |
| 주기적 실행 | **Scheduled Trigger** | 별도 인프라 없이 주기적 감시 |
| PR 관리 | **GitHub CLI** | 브랜치/커밋/PR을 CLI로 자동화 |
| 코드 리뷰 | **GPT Codex** | PR 생성 시 자동 리뷰 |
| 배포 | **Docker + CI/CD** | dev 머지 → 자동 빌드/배포 |

## 현재 한계 및 로드맵

### 현재 한계

| 한계 | 원인 | 영향 |
|------|------|------|
| 5~10분 감지 지연 | Polling 방식 (주기적 API 조회) | P0 에러에 즉시 대응 불가 |
| 승인 대기 | 개발자가 직접 확인 필요 | 부재 시 자동 수정 지연 |

### 로드맵

| 단계 | 개선 | 효과 |
|------|------|------|
| Phase 2 | Webhook 기반 실시간 감지 | Polling 제거, 즉시 대응 |
| Phase 3 | 메신저 버튼으로 승인/거부 | 세션 없이 승인 가능 |
| Phase 4 | 반복 에러 패턴 학습 → 자동 해결 | 승인 없이 완전 자동화 |
